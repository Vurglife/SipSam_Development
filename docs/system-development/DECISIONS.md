# Decisions

| Date | Decision | Reason | Alternatives Rejected | Revisit Trigger |
| --- | --- | --- | --- | --- |
| 2026-05-21 | VurgLife friend messages use a dedicated `friend_messages` inbox table plus `/api/friends/messages` routes, not the legacy room `messages` table. | Friend-to-friend messages need inbox/sent/delete/unread behavior without mixing with game room chat records. | Reuse the existing `messages(room_id, user_id, username, content)` table. | If a unified platform messaging service replaces both room chat and friend DMs. |
| 2026-05-20 | Blackjack invite receivers join with an explicit guest role and are never host candidates; only non-guest seats can start a waiting multiplayer table. | Accepting an invite before the sender is recognized in the room must not make the invitee host. Guests should see only Cancel in the pre-game lobby and can still occupy an open seat if the table is already active. | Continue inferring host from first occupied seat only; rely on client-side hiding of the Start button. | If Blackjack adds persistent room ownership records or a full server-side lobby service. |
| 2026-05-20 | Blackjack includes plain fixed-bet `$500` and `$1,000` tables alongside `$100`, one `$50,000` VIP, `$100,000` Elite, and `$500,000` Celestial. | Product owner wants the mid-low Blackjack tables back, but not branded as VIP or called out as Standard in the dashboard. | Keep only `$100` before VIP; or reintroduce lower VIP tables. | If table economics are centralized in a shared Blackjack config module. |
| 2026-05-20 | Blackjack has exactly one VIP table at `$50,000`; lower `$1,000`, `$5,000`, `$10,000`, and `$20,000` VIP tables are removed. | The product direction is fewer, clearer premium Blackjack rooms: Standard, VIP, Elite, and Celestial. | Keep a VIP ladder for every intermediate Blackjack tier. | If the product owner asks for intermediate non-VIP Blackjack tables later. |
| 2026-05-20 | Blackjack premium dashboard tables mirror SipSam's VIP/Elite/Celestial visual language, and premium table keys are synchronized across dashboard, platform wallet API, and Blackjack WS server. | Enhanced table cards must not advertise rooms that the wallet route or WS server cannot open. Elite uses `$100,000`; Celestial uses `$500,000`, matching SipSam's high-tier anchors. | Apply CSS-only labels to existing Blackjack tables. | If Blackjack tier economics change or a shared cross-game table-config module is introduced. |
| 2026-05-20 | Blackjack client source of truth stays in external `blackjack-client/game.js`, `style.css`, `vl-card.css`, and markup in `index.html`; recovered inline style/script tails are removed. | The recovered inline tail was malformed and made menu/chat/layout fixes hard to reason about. External files support cache busting and focused validation. | Keep appending recovered inline patches to `index.html`. | If a build step or bundler replaces the current static-client model. |
| 2026-05-20 | Blackjack card backs use `blackjack-client/backVurgLife.png`, copied from the SipSam card back asset. | The product owner requested the VurgLife back graphic and Blackjack should visually match established VurgLife games. | Reference the poker-client asset across folders or keep the generic blue back. | If a shared assets folder is introduced for all games. |
| 2026-05-20 | Blackjack uses a duplicated center active-hand focus while still rendering cards at the player's seat. | Mobile play needs readable card values and action controls close to the active hand, while table context still requires cards at each seat. | Only enlarge seat cards, or move cards away from the seat entirely. | If 375px live testing shows overlap or center focus blocks needed table context. |
| 2026-05-15 | Apply the `system-developer-analyst` project template to PokerProject. | PokerProject is a multi-app, multi-session platform where durable memory, validation, and lessons reduce cost and repeated discovery. | Continue relying on chat history and scattered handoff documents only. | If the template becomes too heavy or is not being used during real work. |
| 2026-05-15 | Place the project memory in `docs/system-development`. | The repository root already contains many handoff and recovery files. A dedicated docs folder is cleaner and easier to find. | Add `BRIEF.md`, `DESIGN.md`, and related files directly to the root. | If the team standardizes on root-level planning files later. |
| 2026-05-15 | Do not change application code during template application. | The current task is to apply the project operating system, not modify game behavior. | Combine project setup with feature changes. | When a specific PokerProject development task is requested. |
| 2026-05-15 | Add a project-local Claude Code setup instead of starting with a reusable plugin. | Project-local `CLAUDE.md`, skills, agents, and settings are easier to validate in PokerProject before packaging reusable workflows. | Build a global Claude plugin first. | When the PokerProject setup proves stable and should be reused across multiple projects. |
| 2026-05-15 | Let Claude Code specialist agents make scoped edits when requested. | Mitstar wants Claude Code to handle implementation directly to reduce human error. Safety should come from project rules, validation, and permission settings, not read-only agents. | Keep all specialist agents read-only. | If agents make broad or unsafe edits, tighten individual agent permissions. |
| 2026-05-15 | Allow Claude Code to stage and commit current-task changes after validation. | Prior uncommitted work was lost; committing meaningful changes is part of the recovery-safe workflow. | Force all commits to be manual. | If branch strategy changes or commits need review gates. |
| 2026-05-17 | Use a project-scoped read-only SQLite MCP server for VurgLife analysis. | It unlocks schema and reporting workflows while keeping `vurglife.db` immutable from Claude Code. | Use direct DB edits or a write-capable community SQLite MCP server. | If analytics move to a separate reporting database or read replica. |
| 2026-05-17 | Package the proven PokerProject Claude setup as `vurglife-dev-os`. | Agents, skills, hooks, and MCP tooling should be portable without relying on old chats. | Keep all workflow assets project-local only. | If plugin validation or install behavior changes in Claude Code. |
